# M0 — Faithful baseline: execution plan

Status: ACTIVE. Reproduce upstream busytex's existing build — its pinned
TeX Live 2023, unchanged — inside our own pinned container. No
modernization, no TL 2026, no API work: M0 exists to prove the toolchain
before we change it (DESIGN.md §9; PROMPT.md M0).

## Acceptance checks (PROMPT.md, verbatim targets)

- (a) `make artifacts` produces the engine wasm/js, worker/pipeline glue,
  formats, and at least one data bundle from a clean container.
- (b) A second clean build produces identical sha256s.
- (c) The `demo/` page loads those artifacts and compiles a hello-world
  document to a valid PDF in Chromium via Playwright.
- (d) THIRD_PARTY_NOTICES.md covers everything vendored.

## Ordered work list (one checkbox ≈ one reviewable commit)

- [ ] **1. Container.** `build/toolchain/Dockerfile`: `ubuntu:22.04` by
  digest, `linux/amd64`, emsdk at 3.1.43 (emsdk repo pinned by commit),
  build prerequisites mirroring upstream CI. Smoke check: `emcc --version`
  inside the built image. Record image build args in pins.lock.
- [ ] **2. Pins + fetch.** `build/sources/pins.lock` (original,
  machine-readable format documented in `build/sources/README.md`) and
  `build/sources/fetch.sh`: download every source named in the lock into a
  local cache (outside the repo tree), verify sha256, refuse anything
  unpinned. Mirrors the split-ISO caching upstream CI relies on.
- [ ] **3. Vendor upstream machinery.** Vendor the busytex build machinery
  (Makefile, build scripts, worker/pipeline JS glue needed by the demo) at
  the pinned commit into `build/upstream/busytex/`, each file with a
  provenance header; full-tarball sha256 in pins.lock;
  THIRD_PARTY_NOTICES.md entry (MIT, commit, README license statement).
  `build/upstream/` is an M0-only staging area — M1 dissolves it into
  `build/engines`/`build/formats` per DESIGN.md §4. In the same commit,
  add a one-line note to DESIGN.md §4 recording this staging area
  (CLAUDE.md: deviations are recorded in DESIGN.md, never silently).
- [ ] **4. `make artifacts`.** Top-level make target that runs the vendored
  upstream build inside the container against the cached sources and lands
  engine wasm/js + formats + ≥1 data bundle in `dist/` (git-ignored).
  Expect multi-hour builds and TeX-toolchain breakage; log every
  failure→fix in docs/LOG.md as it happens. Acceptance (a).
- [ ] **5. Reproducibility.** `build/repro-check.sh`: two clean-container
  builds, sha256 manifest diff. Fix nondeterminism at the source
  (`SOURCE_DATE_EPOCH`, stable file ordering, locale/umask pins).
  Acceptance (b).
- [ ] **6. Demo + Playwright.** `demo/` static page loads the artifacts and
  compiles hello-world to PDF; Playwright asserts a valid PDF in Chromium.
  CI `build` workflow gains a guarded job that skips cleanly when
  artifacts are absent (keeps CI green without a multi-hour CI build; the
  full CI build lands with M4 release engineering). Acceptance (c).
  *Explicit deviation from PROMPT.md working rule 3 ("acceptance checks
  pass in CI"): CI execution of the M0 build is deferred to M4's
  release/reproducibility gates per DESIGN.md §9; M0 acceptance runs
  locally and is recorded with artifact hashes in docs/LOG.md.*
- [ ] **7. Notices audit.** THIRD_PARTY_NOTICES.md complete for everything
  vendored; license-audit workflow verifies provenance headers under
  `build/upstream/`. Acceptance (d).
- [ ] **8. Acceptance run.** Execute (a)–(d) end to end, record results and
  artifact hashes in docs/LOG.md, mark M0 complete there and here.

## Risks / standing decisions

- **arm64 host vs amd64 pin.** Dev machine is Apple Silicon; upstream CI
  and our future CI are x86_64. Decision: pin `--platform linux/amd64`
  (Rosetta) so local and CI artifacts are comparable. If emulated build
  time proves unworkable, revisit with a documented deviation and an
  explicit check that wasm outputs are host-arch-independent.
- **ISO URL rotation.** Yearly CTAN URLs die when the next TL ships; only
  historic-archive mirrors are pin-safe. fetch.sh must never encode a
  "current" URL.
- **Upstream has no container.** The Dockerfile is original work (ours,
  MIT); only the build machinery it invokes is vendored busytex.
- **Provenance discipline.** Research around WASM TeX inevitably surfaces
  GPL wrappers; per DESIGN.md §2 their sources are never opened. Note any
  such encounter in the commit message of the work that avoided it.
