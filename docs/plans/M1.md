# M1 — Runtime v1 (MVP core): execution plan

Status: ACTIVE (opened 2026-07-22 after M0 closure; loop runs through M3
per the standing push-through directive). Implement the `wasmtex` npm
package exactly per DESIGN.md §5 — the original, embedding-first API —
over a correlated worker protocol, and migrate the demo off the vendored
glue. This is the project's declared core (§9 revision).

## Scope (as amended 2026-07-22)

- **XeTeX-first**: `engine: 'xetex'` fully supported end to end
  (engine pass → `xdvipdfmx` → PDF). `'pdftex'` exposed only if it costs
  nothing extra once sequencing exists (it shares the binary; simpler
  pipeline). `'luatex'`: enum value reserved, unimplemented, rejected
  with a clear error.
- Single bundle world: `texlive-basic` preloaded via a **data-driven
  asset list** (a minimal `assets.json` generated from `dist/`), not
  hardcoded names. tlpdb tiering, on-demand resolution, and the
  integrity manifest are M4 — but nothing in M1 may bake in asset names
  that M4 or the M2 rebase would have to chase through code.
- No network, no DOM, classic workers, no browser-storage dependence
  (§5.2, §10). Cancellation and correlation are correctness features.

## Rebase-proofing rules (established 2026-07-22, binding on every item)

1. Asset inventories are **data** (assets.json), never code constants.
2. The diagnostics parser is developed against **fixtures captured from
   the current artifacts' real transcripts**; fixtures regenerate at
   every rebase; the conformance corpus re-runs them.
3. **No TL version strings asserted in runtime code** — banners are
   asserted only in build-side gates, updated with the pin.

## Acceptance (PROMPT.md "Runtime v1", as amended + M0 carry-forwards)

- The §8 runtime unit tests pass, including, by name:
  (i) a late message carrying a cancelled/foreign jobId can never
  resolve a newer job; (ii) `cancel()` kills in-flight work (worker
  terminated) and the next job starts clean on a fresh worker; (iii) a
  deliberately broken document yields structured diagnostics with
  file/line; (iv) the diagnostics parser passes its fixture corpus.
- Demo runs on the `wasmtex` runtime (vendored glue no longer loaded by
  the page); Playwright smoke stays green and gains a **content-level
  PDF assertion** (M0 gap #1: extracted text or an uncompressed-stream
  probe — not just structure).
- A typeset-path integration test exists beyond the demo (M0 gap #2):
  compile a fixture document through the runtime under automation.
- CI `runtime-tests` workflow actually runs the unit tests (replaces
  the green-by-construction skeleton) and is green on the pushed HEAD.

## Risks / standing decisions

- **Worker/wasm testability.** Unit tests must not require a browser:
  protocol, sequencing, diagnostics, and client-side correlation are
  pure modules tested in node; the wasm compile path is covered by a
  node-driven integration test (engine under node, as the execution
  gate already proves viable) plus the Playwright demo. Browser matrix
  stays M5.
- **Vendored glue stays in dist/** (provenance + upstream parity) even
  after the demo stops loading it; our runtime replaces its *role*.
  The glue's message contract (journaled at 6N) is the behavioral
  reference, but the §5 API is an original design — no API shapes are
  copied (DESIGN §2.4; busytex is MIT so this is posture, not
  obligation).
- **pdfTeX "near-free" test:** if items 5–7 need any pdftex-specific
  branching beyond driver argv and format choice, defer pdftex to
  post-MVP rather than grow scope.
- **Node for tests:** host node 24 (documented-not-pinned,
  native-host.md); CI uses actions/setup-node pinned major. Hard
  pinning → M3.
